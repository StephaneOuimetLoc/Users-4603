eSuite - Shopper Level Table

eSuite > Customers > Master > General List

Information

The Shopper level table contains special discounts or price levels related to customers. Each record in the Shopper level table must be linked to a price level to establish the item prices used for the group of customers.

The shopper level 1 is the default level used for the regular customers, when no customer is entered in the transaction.

Shopper levels

The Shopper levels list displays the shopper levels available for all customers. Shopper levels contain special pricing information. They are used to group the customers for particular promotions.

In the bottom menu:

  • Edit provides access to the Shopper level table.

The Shopper level field

The Shopper level (Shop. level) field contains the shopper level number. It is a required field, i.e. new shopper level records must have a valid entry in the Shopper level field for the record to exist. It is used as a key field to uniquely identify records in the Shopper Level List.

The shopper level 1 must exist in the database since it is the basic level: if no customer is entered for a transaction, the shopper level 1 is used by default.

Back to layout

The Price level field

The Price level field is the link between the shopper level to which a group of customers belongs and the price level that establishes the item prices used for that group of customers.

The price level is not a required field. If no price level is specified, the price level 1 is used by default, unless another base price level is set up (System.ini). In this case, the set up price level is used. See the reseller for more details on the set up price levels.

The Price level field must be set to 0 when a cost plus percentage is defined.

NOTE:
Starting v350x, Application.ini[TARGET_TRS] is used to determine which target to use for each item table.
Application.ini[TARGET_TRS]PRICE_TAB='PAL' will force the Host to use the PAL record.
Application.ini[TARGET_TRS]PRICE_TAB='@WIZGET(TARGET)' will force the Host to use the Global Store Number record.
Application.ini[TARGET_TRS]PRICE_TAB=ISNULL((SELECT TOP 1 SUB_PRI.F1000 FROM PRICE_TAB SUB_PRI JOIN LNK_TAB SUB_LNK ON SUB_PRI.F1000=SUB_LNK.F1000 JOIN LNK_TAB SUB_LNK2 ON SUB_LNK.F1056=SUB_LNK2.F1056 AND SUB_LNK.F1057=SUB_LNK2.F1057 AND SUB_LNK2.F1000='@WIZGET(TARGET)' JOIN STO_TAB SUB_STO ON SUB_LNK.F1000=SUB_STO.F1000 WHERE SUB_PRI.F01=PRI.F01 AND SUB_PRI.F126=PRI.F126 ORDER BY SUB_STO.F1937 DESC),'PAL'). That will use the proper target set for the Global StoreID selected.

The Fallback field

The Fallback field links the shopper level to a fallback price level, used when the linked price level does not exist for the items.

The system looks up the items at the price level specified in the shopper price level. If the system does not find the items at this price level, it cascades to the price level specified in the fallback price level. If the system does not find the items at this second price level either, it uses the price level 1 (default price level) or the base price level. However, it is possible to change this order by specifying a best price method.
See the Best price method section for more details.

The Base field

The Base price level (Base) field is a link between the shopper level to which a group of customers belongs and the base price level that establishes the item prices used for that group of customers when the price level does not exist for the items. The Base field is used for the prices not to default to price level

1. The system looks up the items at the price level specified in the shopper price level. If the system does not find the items at this price level, it cascades to the price level specified in the fallback price level. Then, if the system does not find the items at this second price level, it uses the base price level. However, it is possible to change this order by specifying a best price method.
See the Best price method topic for more details.

If no base price level is specified, the price level 1 is used as base price level (unless a different base price level has been defined in the System.ini file).

The base price level may be specified for the whole system instead of only for specific shopper levels (System.ini file). See the reseller for more details.

    The Coupon multiply field

    The Coupon multiply (Cpn multiply) field contains the multiplier coupon factor used to multiply the value of the vendor coupons. For example, the multiplier can be used to double, triple or increase by half, one and a half, etc. the value of the vendor coupons.

    The coupon multiplier also applies to the vendor electronic coupons. By default, the add-on values (the amounts over the vendor coupons values) are considered as store coupons.
    See the Preferred shopper - double coupons topic for details on the vendor coupons management.

    The Add value field

    The Add value field contains a fixed amount that increases the value of the vendor coupons. For example, an amount of $0.50 can be added to the vendor coupons as a promotional discount.

    The value entered in this field must be a numeric one and must contain a decimal point as a separator between dollars and cents. There is no fixed limit maximum amount.

    The fixed amount also increases the vendor electronic coupons values. By default, the add-on values (the fixed amounts) are considered as store coupons.
    See the Preferred shopper - double coupons topic for more details on the vendor coupons management.

    The Point programs field

    The Point programs field contains the identifying characters of the point programs that the customers linked to the shopper level are members of.

    The system can manage up to 35 point programs. They are identified by characters from 1 to 9 and from A to Z: A corresponds to point program 10, B corresponds to point program 11, up to Z that corresponds to point program 35.

    Point program 1 is the main point program. All the others are secondary point programs. They all work independently, according to their own methods, with no interaction between each other. The Point programs field allows multiple values (do not insert any characters).

    See the Preferred shopper - Points and Preferred shopper - Points redemption topics for details on the points and the point redemption management.

    The Penny/point field

    The Penny/point field contains the number of pennies needed to get a point. It is also called the point rate.

    It is used for item, sub-department and department sales, but it can also be used to give points on any function, like tenders, sub-total, etc. See the Point method and the Preferred shopper - points topics for more details on the points management.

    NOTE: Depending on the actual point method used, the Points field located in the Price table may have priority over the Penny/Point field, i.e. if a number of points is entered in the Points field for an item, these points are applied instead of the point rate for that item. This functionality is useful when you want to exclude specific items from being counted in the total of points given (Points field = 0). If the Points field is empty, the point rate is applied for that item.

    Back to layout

    The Expiration days field

    The Expiration day (Exp. day) field contains the number of days the points are kept in the customer's points balance for the main point program (point program 1).

    The points expiration is done using a script. A scheduler job can be created so the expiration runs automatically. All unused points at the expiration's date (today's date - number of days specified) are expired (an adjustment transaction is performed).

    NOTE: See the Preferred shopper - Points redemption topic for details on the point redemption management.

    No best price method The No best price method (default value) specifies that the system should follow the default process, i.e. the system should first look up the items at the shopper price level. If the system should not find the items at this price level, it should cascade to the fallback price level. Then if the system should not find the items at this second price level, it should use the base price level (default price level). In the case where there is no base price level defined, the price level 1 is used by default (unless a base price level has been defined in the System.ini file - see you reseller for the details).
    Best of 3 customer levels The Best of 3 customer levels method specifies that the system should look up the items at the three (3) price levels defined in the shopper level (shopper price level, fallback price level, base price level), and take the best price of all three.
    Best of all levels The Best of all levels method specifies that the system should look up the items at all price levels available for each item sold, and take the best price. This includes the cost plus price and the contract price.

    The Point balance over field

    The Points balance over field contains a number of points used as a condition to upgrade the shopper level for the current transaction. The shopper level upgrades to the shopper level defined in the Upgrade to level field if the customer's points balance is equal to or higher than the points balance over number.

    The points balance at the beginning of the transaction is used to validate the Points balance over value; the points earned during the transaction are not considered. For example, let us say 5000 points are required to upgrade to another shopper level. If the customer starts the transaction with 4990 points and gets 20 points in the transaction for a new points balance of 5010 points, the shopper level upgrade will not be performed since the points balance at the beginning of the transaction is under 5000 points.

    NOTE:  For the customer's points balance to be considered, a customer must be entered in the transaction. See the Preferred shopper - Shopper level upgrade topic for details on the shopper level upgrade management.

    The Point spending field

    The Period spending field contains an amount used as a condition to upgrade the shopper level for the current transaction.     The Start section

    The Start field is a time field that contains the starting hour used as a condition to upgrade the shopper level for the current transaction. The shopper level upgrades to the shopper level defined in the Upgrade to level field if the transaction is performed during the period defined by the starting and ending hours of the corresponding day.
