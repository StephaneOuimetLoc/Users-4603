eSuite - POS Information Table

eSuite > Items > Master > Item list

Information

The eSuite software uses relational tables to store item information. The POS table contains POS sales information such as tax status, discount status, etc. for each item in the Main item table. Some of the POS table information may not apply to all POS systems.

The POS information table can contain multiple records for the same item based on regional differences such as taxability, so that a head office can control regional (target) differences. The eSuite software uses a unique key made up of two fields: the main item UPC and the target. This means that each UPC can have more than one point of sale record in the POS information table, as long as the targets change. The same UPC can have as many different records in the POS Information table as there are unique targets.

Displaying the POS information table

In the Launcher, click the arrow in the Items entry to display its menu and then:

  1. Click on Master: the Start window opens in the Items work area
  2. Click on Item list: the Wizard tab opens in the same window, displaying the Items - General List
  3. Enter the search criteria and click on Execute: results will display in a record grid, in the Items tab of the same window
  4. Select a record and click on Edit on the bottom of the item grid: the option menu is displayed
  5. Select Master tables and then POS master: the POS information edit table opens overlapping the work area

The POS description field

The POS description field contains the description of the item displayed on the POS screen and/or printed on the receipt.

Since some POS systems, such as the ECR systems, can only display/print a limited number of item descriptor characters on the POS screen/receipt, the item descriptor (main item descriptor) may be truncated to fit the POS screen/receipt if it contains more characters than the POS character limit. In this case, it can be difficult to correctly identify the item by its descriptor. The POS description field allows you to control the descriptor characters sent to the POS system.

To enter a POS description, type the POS description of the item in the POS description field while respecting the POS system descriptor character limit (specified in the POS system documentation).

The Sub-department field

The Sub-department field links the item to a sub-department.

In most POS systems, item sales are linked to sale categories for reporting purposes. Because different POS systems use different terminology for sale links, it is important to understand the eSuite terminology and how it relates to the POS system. The eSuite software uses a three-tier sales link level organization:

  1. Item sales are accumulated in sub-departments.
  2. Sub-department sales are accumulated in departments (a department sales total contains sales from one or more sub-departments).
  3. Department sales are accumulated in group sales (a group sales total contains sales from one or more departments).

The eSuite software sales link levels (sub-departments, departments and groups) are defined in the Sub-department and Department tables.

Whether or not the POS system uses the same terms, eSuite requires that items be linked to a sub-department or a first sales link level.

The Divisor field

The Divisor field contains the weight divisor of the item.

Sometimes, item prices by weight can be very high per kilogram or pound. In these situations, it is preferable to use a smaller measure for pricing. When the item is sold, the POS divides the weight by the divisor and the prices accordingly.

The Follow sub-department field

The Follow sub-department (Follow sub) field specifies whether the item follows the linked sub-department sales information programming (checked), or not (unchecked). It is checked by default.

It is possible to program the point of sale information once for a sub-department, and then apply the programming to the items that are linked to that sub-department. This accelerates the item programming process since you do not have to individually set each item's sales information.

When the Follow sub-department field is checked, some fields that may be programmed at the sub-department level get read-only. They all become available when the Follow subdepartment field is unchecked.

NOTES:
The Exclude spending field of the Price table is also included in the fields that are linked to the Follow sub-department field.
The default list of fields can be modified. See the reseller for more details.

The Customer age field

The Customer age field contains the minimum age the customers must be of to buy the item. For example, in areas where stores are allowed to sell beer, wine or alcohol, customers must be of legal adult age for that area to buy these items.

The Customer age may be specified for a whole sub-department or department keys.
If there is no customer age specified at the item level, the system automatically looks for a customer age at the sub-department level ONLY if the Follow sub-department field is checked.

When the customer's age is specified both at the item level and at the sub-department level, the system verifies both values. If these values are different, the customer age specified at the item level has precedence over the customer age specified at the sub-department level.

When a sale is performed at the POS, the cashier is prompted to enter the customer's birth date. The system automatically calculatee the customer's age and compares it to the value ointhe Customer age field. If the customer is under the minimum age required, a warning message is displayed on the POS screen.

NOTES:
 If the operator is a level 5 or higher, this restriction will be bypassed.
See the reseller regarding the possible alternate configurations (System.ini file).

The Stop discount field specifies whether discounts may be applied on the item when it is sold at the POS (unchecked) or not (checked).

It does not allow manual item discounts, proportional discounts, global discounts, customer discounts (shopper level discounts) and sub-department discounts on the item.

The Not for sale field

The Not for sale field specifies whether the item can be sold at the POS (unchecked) or not (checked).

When the Not for sale field is checked, the item remains in the database, but cashiers are not able to sell it at the POS. This is useful for out-of-season items that will be available again later. Cashiers will not be able to sell the item by manual code entry error, and skewing the inventory reports.

NOTE: If a scalable item is set as 'Not for sale' and deployed, the code SPID will be sent to the scale which is an item delete. The item and its information will be removed from the scale.

The Not in net field

The Not in net field specifies whether the item sales are added to the store net sales (unchecked) or not (checked).

When the Not in net field is checked, the POS will treat the drawer amounts for balancing purposes, but the sales amounts will not appear in the net sales. For example, a PLU code can be assigned to a consigned item, which is not considered as a sale for the store.

NOTES:
 For this field to work, the item must not be programmed to follow the sub-department settings.
The Not in net field is often referred to as "hash sales".

The Store coupon field

The Store coupon field specifies whether the item is an electronic store coupon (checked) or not (unchecked).

Store coupons are used by retail businesses to promote sales. They can be coded and scanned, or manually entered. A store coupon is negative within a POS transaction.

By default, an electronic store coupon directly affects the amount of the items in the transactions like any item discount, so it affects the item sales. It also borrows the taxability of the items. The discount of an electronic store coupon is assumed by the store.

NOTE: For more information on the electronic store coupons, see the Electronic coupons programming topic.

Back to layout

The Vendor field

The Vendor field specifies whether the item is an electronic vendor coupon (checked) or not (unchecked).

Vendor coupons are distributed by manufacturers and vendors to promote sales. They can be coded and scanned, or manually entered. A vendor coupon is negative within a POS transaction.

By default, an electronic vendor coupon is added as a new entry (registration) in the transactions, so it does not affect the item sales, and the taxability of the items is not borrowed. The discount of an electronic vendor coupon is usually assumed by the manufacturer or the vendor.

NOTE: For more information on the electronic vendor coupons, see the Electronic coupons programming topic.

The Multipliable field specifies whether coupon values may be multiplied at the POS for the item (checked) or not (unchecked).

On some vendor coupons, manufacturers do not allow double coupons, that is the store cannot multiply the value of these vendor coupons or give an add-on amount as an additional discount for the customer.

The Rain check field

The Rain check field specifies whether rain checks can be generated for the item (checked) or not (unchecked).

Occasionally, retail outlets run out-of-stock for a promotional item, but would like that the customers be able to purchase the item later (after restocking) at the promotional price, even if the promotion has ended. In this case, print coupons (or rain checks) may be issued for the item, allowing the customer to use them later to obtain the promotional price.

NOTE: For the rain checks, or the electronic coupons, to be printed, a form can be defined in the Ticket field of the POS table; in this case, this form is used instead of the default one (see the How to program rain checks topic or How to program printed electronic coupons topic for more details).

The URL field

  • The URL field contains special parameters (also called 'alpha parameters') used when the item is sold.
  • The URL field can contain more than one parameter, separated by spaces.

Examples of alpha parameters for the URL field:

KEYWORD DESCRIPTION
WIT=REG
WIT=NEWREG
WIT=NEWCPN		 These alpha parameters are used on electronic coupons to change their default behavior.

It is useful to change the behavior of specific electronic coupons instead of all of them. See the reseller to get the details of possible behaviors.
POINTx= This alpha parameter is used to give (or do not give) points on specific items for the points program x.

Possible values are:

  • yyy: defines a rate to calculate the points based on the price of the item (e.g. POINT1=100 will give 1 point per $1.00)
  • Y: tells the system to use the rate defined in the shopper level the customer is linked to (e.g. POINT1=Y)
  • N: tells the system to do not give points on this item (e.g. POINT1=N)
  • Fyy: defines a fixed number of points to give on this item (e.g. POINT1=F10)

This alpha parameter can be used to multiply points on specific item (meaning additional points are given on this item).
