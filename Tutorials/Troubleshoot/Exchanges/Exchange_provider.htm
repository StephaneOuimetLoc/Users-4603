Exchange Provider - Overview

eSuite > System > Exchanges

Information

The Exchange provider manages and carries out the important task of controlling the exchange of information between the back office server and the front end workstations or the head office and the stores.

The process is similar in concept to the electronic mail, with inbox and outbox folders for each store, server and terminal/lane configured for the communication exchange. The information exchange process requires several steps for sending and receiving data, including checking exchange mailboxes and the scheduler list for activities.

Monitoring exchange messages

One of the most important daily store tasks is monitoring the results of exchange communications in the Messages section of the eSuite System module. If errors occur during the data transfer, the system writes an urgent message that includes the details of the error. Successful communications are noted as normal messages. If errors occur during the communication process, the system will set the status of this terminal to offline, putting a hold on normal communications. The terminal stays offline until its status is manually reset to online.

Displaying the exchange provider

In the Launcher, click the Exchanges entry:

  1. The Start window opens in the Exchange work area.
  2. Click on Exchange provider: the exchange provider opens in the Provider tab.

Content

Here is a description of the Exchange provider fields and functionalities:

FIELD DESCRIPTION
Node The Node field contains the six-digit node number (store and terminal number).
Status The Status field contains the last status of the node (online or offline).
Fails The Fails field contains the number of failures (if the system has not been reset since).
Last fail The Last fail field contains the date and time of the last failure (if the system has not been reset since).
Description The Description field contains the description of the node.
Server The Server field displays the server number of the selected node.
Send driver The Send driver field contains the sending information of the exchange driver needed for the communication and the transfer of information between the node and the backoffice server.
Receive driver The Receive driver field contains the receiving information of the exchange driver needed for the communication and the transfer of information between the node and the backoffice server.)
Menu The menu offers the following options:
  • The Stop option stops the communication for the selected node.
  • The Stop all option stops all nodes listed in the Exchange provider table.
  • The Restart option restarts the communication with the selected node.
  • The Restart all option restarts all offline nodes listed in the Exchange provider table.

Stopping/Restarting Communications in the Exchange Provider

The Exchange provider uses a programmable clock cycle to determine the frequency of the terminal's activities. As a user, there are only a few reasons to check the Exchange provider. For the most part, after it has been configured, the Exchange provider acts on its own, without any user intervention.

However, there are occasions that require a manual intervention, like communication issues. If a POS terminal (front end workstation) is removed on purpose from the network, for example due to maintenance, the Exchange provider must stop trying to communicate with this missing terminal until it has been returned to the network. If a temporary communication failure has been corrected, the user may need to force the Exchange provider to restart the communication.

Stopping communications

Occasionally, it might be necessary to stop the communication between a node and the back office server, such as if a node is removed from a store.

To stop the communication with a node, follow the steps below:

  1. In the Exchange provider table, select the node for which communication must be stopped and click on the Menu button in the lower part of the table.
  2. Click Stop to stop the communication for the selected node.
    The status of the node changes to "HOLD".

Restarting communications

If a node has been placed offline on purpose, the status of the node displays "HOLD" and no messages are posted in eSuite. However, if the system loses communication with a node for any other reason, the status of that node is displayed as offline in the Exchange provider table. When a node is offline, the urgent messages must be checked to find the source of the error. Once the offline issue of the node is resolved, communication between the node and the back office server can be manually restarted in the Exchange provider.

NOTE: The status of Nodes/terminals for which communication has been stopped on purpose will still be on "HOLD", even if users log back into eSuite.

To manually restart the communication with a node, follow the steps below:

  1. In the Exchange provider table, select the node for which communication must be restarted and click on the Menu button in the lower part of the table.
  2. Click Restart to restart the communication with the selected node.
    The status of the node changes to "ONLINE".

Data updates once communication is established

Information exchange for any job, communication or any other event that was not successful remains in the target nodes mailboxes, so that when errors are resolved, the information is sent to the node once the communication is re-established. For example, a node is down for several days due to errors or maintenance; once the issue is resolved or the node is repaired, it will receive all price changes since the last successful communication, in the order they were sent. This will successfully update the node.

Status of the node mailboxes can be checked using the MailBoxes menu in the Messages section of System module in eSuite.

Communication exchange error messages

Every failed communication produces an urgent message in the messaging system. When the Exchange provider tries to communicate again with the node and fails subsequent times with the same error, the system adds the error in the same error message. In case when the error changes or another node is involved, the system will create additional messages for those errors.
