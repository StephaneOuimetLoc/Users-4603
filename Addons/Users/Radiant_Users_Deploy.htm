Moadule - Radiant Interface - Users Deploy

Radiant Unmanaged

Overview

When executing a deployment, new, modified and/or deleted data is stored in a file that is sent over a network from the LOC SMS server to the Radiant server. Please refer to the sections below to acquire the knowledge on functionality between the two (2) systems.

This portion of the document describes the auxiliary functions used between the LOC SMS back office server and the Radiant interface.

Auxiliary Deployment

The deploy_auxiliary process is only used to send sub-departments, departments and bottle link data to the Radiant back office server.

It is important to know that if changes are made and deployed for the auxiliary tables, it is required to make a minimal change to an item and execute deploy changes to send the items again. If this is not done, the items will not get updated with the new auxiliary settings.

Sub-Departments and Departments

By default, sub-departments are sent with deploy auxiliary. This can be changed to send departments instead.

NOTE: If departments are meant to be sent instead, this needs to be setup during the installation process. Speak with the technician performing the installation.

Bottle Links

Bottle Links are considered items in Radiant. In order to differentiate bottle links from regular items, a set prefix for 'BottleLinkPattern' will be configured by the technician performing the install in the XML file. Create the bottle link as normal. Upon deploying auxiliary, the prefix will be added to the bottle link.

Ex.: Bottle Link prefix = 9??????????????, Bottle Link number = 10. When deployed to Radiant back office, will appear as an item with code 900000000000010.

NOTE:Bottle links must be deployed before sending an item with a bottle link to Radiant. If an item is sent containing a bottle link, but the bottle link list was never sent, the bottle link will not appear with the item.

To execute the deploy auxiliary process, follow these steps:

  1. In eSuite\man, select Deploy auxiliary from the Tools\Deploy left hand side menu.
  2. Select DRAD as Target, the Action desired and the Deploy type, click EXECUTE.

Item Deployment

Overview

This portion of the document describes the item functions used between the LOC SMS back office server and the Radiant interface.

Item Creation

For information on creating and deploying items, please refer to the Creating a new item document.

Car Wash Items

In order for an item to qualify as 'Car Wash' a few parameters must be set.

  • Type needs to be set to 10 in the POS Table. (Image pending)
  • NACS code needs to be set and correspond to the correct value, see table below:
    Type Coded NACS Code
    Express (Basic) 1 102
    Super 2 800
    Deluxe 3 801

EAN8 Items

In order for an item to qualify as 'EAN8', simply create an 8-digit code in SMS.

In SMS, the code will be padded with zeroes (0) to equal 13 digits. When the item is deployed to Radiant, the padded zeroes (0) will be stripped, no check-digit is added.

Item Deletion

For information on deleting items, please refer to the Delete an item document. (Under development)

NOTE: When deleting an item in LOC SMS and deploying to Radiant, the item will still be visible on the Radiant back office server but it will be disabled. Therefore, it cannot be sold at the POS.

Mix & Match Deployment

Overview

This portion of the document describes the mix & match functions used between the LOC SMS back office server and the Radiant interface.

Mix & Match Deployment

The deploy_changes process is used to send mix & match data to the Radiant back office server.

The Mix & Match ID has to be created as an auxiliary first, which then needs to be linked to the necessary items.

The Mix & Match promotion that will be deployed and available at the POS is the mix that corresponds with the active price. The max package price and quantity of all items with the same Mix & Match ID is sent.

NOTE: If the Mix & Match ID is not created in the Auxiliary table prior to linking it to an item, the Mix will never attach to the item properly and will not be given at the POS.

Mix & Match Deletion

The user can remove a mix from an item. Upon the next deployment, the mix will be deleted from the item on the Radiant back office server.

When all items containing this mix have had the mix deleted, the Mix & Match list will be deleted from the Radiant back office server as well.

Combo Deployment

Overview

This portion of the document describes the combo functions used between the LOC SMS back office server and the Radiant interface.

Combo Deployment

The deploy_changes process is used to send combo data to the Radiant back office server.

Combos in Radiant are considered as electronic coupons in SMS. The coupon link requires a Promo ID sales code (PID) and quantity in order to deploy correctly to Radiant.

 The combo deployed to, and available at the POS, is the combo that corresponds with the active price. The coupon needs to be linked to the necessary items with a PID.

Price Methods

  • The price field has 2 functionalities;
    • Final price for a coupon bundle (CB method) once the coupon link quantity is reached.
    • Dollar off amount to the sub-total of the transaction once the coupon link quantity is reached.
  • The discount percent field will give a percent value off the sub-total of the transaction once the coupon link quantity is reached.

WARNING: Price methods cannot be combined. If multiple are selected, the combo will not be deployed.

Limited Quantity

It is possible to limit the quantity of times the combo/discount is given to the customer in a single transaction. This can be achieved by entering a numeric value in the limited quantity field, and adding the Limit by Transaction (LT) method. The value entered represents how many combos may be awarded.

NOTE: If the LT method is not present, the combo will be given endlessly.

Combo Deletion

The user can remove a PID from an item. Upon the next deployment, the PID will be deleted from the item on the Radiant back office server.

When all items containing this PID have had the PID deleted, the combo list will be deleted from the Radiant back office server as well.

Operator Deployment

Overview

This portion of the document describes the operator functions used between the LOC SMS back office server and the Radiant interface.

Operator Deployment

The deploy_changes process is used to send operator data to the Radiant back office server.

The operator record requires it to be created with a number, level, password, first and/or last name and active/inactive status. Operator birth date is optional.

NOTE: The level selected will correspond to the security level mapping set in the ApiXchRadiant.xml, which is configured by the reseller during installation.

Operator deletion

The user can remove a value from an operator record. Upon the next deployment, the record will be updated on the Radiant back office server.

When deleting an operator in LOC SMS and deploying to Radiant, the operator record will removed from the Radiant back office server.

NOTE: It is suggested, as a best practice, to keep the operator record as inactive for a minimum of 5 to 7 years before removing it entirely in case of government auditing.
