eSuite - Interdepartmental Sales

eSuite > Items > Inventory

Information

To be revised once this feature is available in eSuite.

The Deli and Bakery departments often use items from the other store departments to produce recipes. Regarding the inventory, it is important to transfer the responsibility of the consumed items from their original departments to the Deli and Bakery departments.

If this is not done, the departments from which the items come will be short in their inventory and the Deli and Bakery departments will generate revenues without any cost of goods. The inter-department sales feature has been created and implemented to sell items to a particular department from any POS or backoffice terminal.

Each department as a customer

Each department to which it is possible to sell items must have a customer defined in the system. The customer number used for these departments should not be too long, usually a 3-digit number. For the system to understand that the sales performed for that customer is an inter-department sale, a link to the Inter-department sales function (547) should be defined in the customer's account. This is done by entering 547 in the Adding function field located under the Miscellaneous tab of the Customer table.

 

Since the inter-department sales are made at the POS using customers, a specific shopper level has to be assigned to them, defining the price at which the items are sold (cost, cost plus, retail with and without discounts, etc.).

Different item codes (UPCs) for the same item

To move the inventory from one department to another, the item code (UPC) of the product needs to change so it could be assigned to a different department. To be able to make inter-department sales, a code used to transpose the inventory when the original code is sold needs to be set up. This code has to be defined in the Inter-department code field of the Main item table, for each item that can be sold to another department. It must contain the letter C at the place where the customer number of the department should go.

In the example below, the item code for bananas is 0000000004011. The code to set in the Inter-department code field for this item may be CCC0000004011.

 

When bananas are sold to another department, the letter C set in the code is replaced by the digits of the customer number of the department to which the bananas are sold. As an example, if the bananas are sold to the Deli department for which the customer number is 501, the item code for these bananas becomes 5010000004011. Selling the bananas to the Bakery department, for which the customer number would be 502, will give 5020000004011 as item code for these bananas.

Since there are new item codes for this kind of items, all these item codes should exist in the database. Following the same example, the item code 5010000004011 should be created in the item database and linked to the Deli department; the item code 5020000004011 should also be created and linked to the Bakery department, and so on. Finally, since these new item codes will be used only to produce prepared foods by these departments, they should be flagged as Not for sale.

   
   
 
 

When they produce recipes that contain bananas as ingredients, the departments will use the item code CCC0000004011 in the Recipe table (5010000004011 for a Deli department recipe, 5020000004011 for a Bakery department recipe, etc.).

Inter-department code missing

By default, the system does not verify if the value in the Inter-department code field is set on an item when making an inter-department sale on that item. If it is not, the inventory is adjusted only on the department that sells the item, and not on the one that is buying it.

It is possible to configure the system so that it verifies if the value is set in the Inter-department code field when the operator is performing an inter-department sale. See the reseller to configure this feature (System.ini/Miscellaneous).

Once this configuration is set up, the operator gets the "Inter-dept code missing" error message at the POS.

Sell items to another department

From the POS, the backoffice or the PDA, the selling transaction should be made to the proper department 'customer' using the item code of the 'main' item.

According to the example of the bananas sold to the Bakery department, the item code 0000000004011 has to be sold to customer 502. The system will move the quantity sold from the original department (Produce, UPC 0000000004011) to the department set in the Inter-department code field (Bakery, UPC 5020000004011).

NOTE: For the inventory to be adjusted in both departments, it must first be initialized LINK to add on all banana items. Otherwise, the inventory will stay at the NO INV status. This means a cost record is needed for all banana items.

When the unit has to change

When a department sells items to another department, the second department may need to change the unit used for these items. It could be the case when the Grocery department sells sugar to the Bakery department. The Grocery department is selling a 500 g bag of sugar. Since the Bakery department will use the sugar in his recipes in small quantities, the 500 g bag of sugar will be used in many recipes. When the inventory is adjusted for the sugar item on these departments, the system will remove a quantity of 1 [bag] from the Grocery department and add a quantity of 500 [g] in the Bakery department (instead of adding a quantity of 1 [bag]).

To be able to make that unit change, the Volume/weight field of the Main item table is used in combination with the Scalable field of the POS table.

So according to the example of the sugar sold to the Bakery department, here is the maintenance setup needed. For the sugar item of the Grocery department, 500 has to be set in the Volume/weight field. For the sugar item of the Bakery department, the Scalable field and the Not for sale field have to be checked.

   
   
   

With this maintenance setup, a quantity of 1 will be removed from the inventory of the Grocery department and a quantity of 500 will be added to the inventory of the Bakery department when a 500 g bag of sugar will be sold to the Bakery department.

Inventory history

Once an inter-department sale is performed for an item, you can use the Inventory history analysis LINK to add to see the details of its inventory history (Inventory history option from the Analysis pull-down in the SMS Pro application).

Example

This example shows how to sell either a wine bottle and glasses of wine and track proper inventory of both (1 bottle of wine = 4 glasses of wine).

Step 1: Create the Wine bottle (750 ml) record with an inter-department code.

UPC for the wine bottle in this example is 0000000000850. The associated inter-department code is CCC0000000850.

 

Step 2: Create the inter-department wine bottle item with split code

The inter-department item and split code are used to correctly track the inventory of the main wine bottle and wine glasses item.

   
   
   

The inter-department wine bottle item is marked Not for sale. The Split code field contains the UPC of the wine glasses item (5010000000849) and the Split quantity contains 4, which corresponds to the number of wine glasses contained in the bottle.
