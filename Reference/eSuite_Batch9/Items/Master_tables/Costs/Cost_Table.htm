eSuite - Cost Table

eSuite > Items > Master > Item list

Information

The eSuite software uses relational tables to store item information. The Cost table contains all cost information, such as vendor, base cost, unit cost, discounts, etc., for the main item. The eSuite software uses the Cost table to retrieve information for purchases and inventory purposes.

Displaying the Cost table

In the Launcher, click the arrow in the Items entry to display its menu and then:

  1. Click on Master: the Start window opens in the Items work area.
  2. Click on Item list: the Wizard tab opens in the same window, displaying the Items - General List.
  3. Enter the search criteria and click on Execute: results will display in a record grid, in the Items tab of the same window.
  4. Select a record.
  5. Click Edit on the bottom of the item grid to display its option menu.
  6. Select Master tables, then Cost master: the Cost edit table opens over the work area.

The Cost table may contain multiple records for the same item, based on regional and vendor differences. This way, a head office can control regional (target) item cost differences. To find a cost record for the main item, the eSuite software uses a unique key made of three fields: the main item UPC, the target and the vendor. This means that each UPC can have more than one cost record in the Cost table, as long as the target and/or vendor change. For the same target, a UPC can have many different cost records as there are unique vendors, and for the same vendor, a UPC can have as many different cost records as there are unique targets.

Because of the multiple records capability, the Cost table can toggle between a simple view and a detailed view.

The simple view consists of the quick reference section and the pricing columns.
The detailed view adds the bottom tab section and the left tab section.

  • The Collapse/Expand left control in the toolbar hides/shows the left tab section.
  • The Collapse/Expand bottom control in the toolbar hides/shows the bottom tab section.

Cost events

The Cost table contains several cost events that allow maintaining the different types of discounts given by a wholesaler or a manufacturer on an item. Each of the cost events contains roughly the same fields, such as Start date, End date, Amount, etc.

The cost events are: Allowance, Discount, Rebate, Break/quantity (cost break). They are displayed in the Deals tab.

Billbacks

Sometimes, vendors can grant retroactive reductions on the purchase price (cost) provided that a certain quantity or value of goods is bought. Theses reductions are called billbacks. They consist of a refunding by the vendor of a part of the sum that was already paid for the purchase of an item.

The eSuite software keeps information on up to four (4) different billback values in a cost record, expressed in dollars or as a percentage.

The bill back values are used for information purposes only. They do not affect the cost amount since the required quantity or value may not be purchased to claim the bill backs.

Billbacks are displayed in the Billback tab.

The Next cost field

The Next cost field shows the next cost event based on the allowance, discount and rebate dates.

However, many factors may affect the unit cost value, such as the allowance, discount and rebate cost events (if they are off invoice) and the different fees (percent %, dollars $ and split case).

The Next cost field is grayed out and used for information purposes only. Its value is updated as soon as the modifications are posted (saved) to the database. The allowance, discount and rebate amounts are cumulative if they are active. If no allowance, discount or rebate is defined, the Next cost field is empty.

NOTE: The fees (percent % and dollars $) also affect the next cost value.

Examples

The regular cost of item #0000000000001 is $45.00.

Case 1

No allowance, discount or rebate is defined.

In this case, the Cost field shows the regular cost $45.00, and the Next cost field is empty.

Case 2

A discount is currently active ($5.00). No allowance or rebate is defined.

In this case, the Cost field shows the regular cost ($45.00) less the discount amount ($5.00), i.e. $40.00, and the Next cost field shows the cost to which item #0000000000001 will return after the discount end date, namely the regular cost ($45.00).

Case 3

A discount is defined for the next week ($5.00). No allowance or rebate is defined.

In this case, the Cost field shows the regular cost ($45.00), and the Next cost field shows the cost that item #000000000001 will be at when the discount becomes active, namely the regular cost ($45.00) less the discount amount ($5.00), i.e. $40.00.

Case 4

A discount is currently active ($5.00), and a rebate is defined for the next week ($10.00), effective immediately after the discount.

In this case, the Cost field shows the regular cost ($45.00) less the discount amount ($5.00), i.e. $40.00, and the Next cost field shows the cost that item #0000000000001 will be at after the discount end date, namely the regular cost ($45.00) less the rebate amount ($10.00), i.e. $35.00.

Case 5

A discount is currently active ($5.00), and a rebate is defined for the next week ($10.00), effective from the last discount active day (crossing for one day).

In this case, the Cost field shows the regular cost ($45.00) less the discount amount ($5.00), i.e. $40.00, and the Next cost field shows the cost that item #0000000000001 will be at when the rebate becomes active, that is to say the regular cost ($45.00) less the discount amount ($5.00) less the rebate amount ($10.00), i.e. $30.00.

The Source field

The Source field is an 8-character field that contains the origin of the current next cost.

The Source field is not editable and used for information purposes only. Its possible values are:

Source Description
BASE when the future cost comes from the Regular cost event.
ALLO when the future cost comes from the Allowance cost event.
DISC when the future cost comes from the Discount cost event.
REBAT when the future cost comes from the Rebate cost event.

NOTE: Even if the Break/Quantity cost event contains active dates, it does not display anything in the Source field because one may not purchase the quantity required to obtain the reduction.

The Vendor ID field

The Vendor ID field links the current item cost record to a vendor. It is a required field, i.e. new cost records must have a valid entry in the Vendor ID field for the record to exist. It is used as a key field, along with the main item's UPC and the Target field, to uniquely identify records in the Cost table.

A main item UPC can be linked to different vendors, creating multiple records for the same item.

To link the item to a vendor, type the vendor's identification code in the Vendor ID field or click on the arrow to select a vendor from the Vendor table.

Records must already exist in the Vendor table for the Vendor table to display choices. The vendor name of the specified vendor is displayed in the Vendor name field. If not, the vendor identification code entered is not valid.

The Unit of measure (UOM) field

The Unit of measure (UOM) field contains the identification characters of the buying format available for the current item.

The Unit of measure field has effects in the purchase orders. The buying formats available in the purchase orders depend on the values of the Unit of measure field.

NOTES:
 The unit of measure may also be specified at the vendor level (Unit of measure field in the Vendor table). In this case, the value defined in the Cost table has precedence over the value defined in the Vendor table.
If the Unit of measure field does not contain any value, the case format is the only buying format available.
A maximum of two buying formats can be specified.

The Split code field section

The Split code field contains the UPC code of the item that is part of the current item. It is used for kits that contain only one (1) item (1 UPC code). It works in combination with the Split quantity field.

For example, a carton of cigarettes (UPC #0000000010000) contains 8 packs of cigarettes (UPC #0000000008000). In this case, the Split code field for the item #0000000010000 (carton of cigarettes) contains the value 0000000008000 (pack of cigarettes).

NOTE: When the base cost of the current item (in the example above, the carton of cigarettes UPC #0000000010000) is modified, the system asks whether to also change the base cost of the split item (in the example above, the pack of cigarettes UPC #0000000008000). Feature available starting in May 2019.

The Status field

The Status field is a drop-down list that contains the identification characters of the item status.

The Status field is used to specify whether the item is accepted in a reception or not, depending on the status type.

The back order status of the item is also managed from the Status field. The Allow BO status indicates that back orders are accepted for the current item. The number of days of the back order life is defined once for all items of the vendor, in the B.O. field of the Vendor table.

NOTE: The default list can be modified. See the reseller for details.

The Inventory field

The Inventory field is a drop-down list that contains the identification letter of the method defining the item cost used for the inventory value of the item.

The following options are available:

METHOD VALUE DESCRIPTION
  (empty) This method uses the method specified in the System.ini file.
This is the default value of the Inventory field.
The default value specified in the System.ini file is Average cost.
Average cost (A) This method uses the average cost of the item.
Last DB cost (C) This method uses the current cost of the item, defined in the Cost table.
Last buying cost (L) This method uses the cost of the last purchase of the item.


Each method has a different effect on the inventory value. See the reseller for details.

The Start date field

The Start date field is a date field that contains the start dates for the cost events, i.e. the dates at which the cost events become effective. The start date is inclusive.

The Allow Start date field contains the start date for the allowance, i.e. the date at which the allowance becomes effective.

The Discount Start date field contains the start date for the discount, i.e. the date at which the discount becomes effective.

The Rebate Start date field contains the start date for the rebate, i.e. the date at which the rebate becomes effective.

The Break/quantity (Brk/qty) Start date contains the start date for the break/quantity, i.e. the date at which the break/quantity becomes effective.
The Break/quantity is not exactly considered as a cost event since the required quantity may not be purchased to claim the reduction. The Break/quantity will give you a reduction if you use the Inventory module to receive goods.

The Allow, Discount and Rebate cost events must contain valid Start and End dates for the eSuite software to recognize the cost reductions.
The Break/quantity does not require Start and End dates.

The End date field

  • The End date field contains the end dates for the cost events, i.e. the dates at which the cost events are no longer effective. The end date is inclusive.
  • The Allow End date field contains the end date for the allowance, i.e. the date at which the allowance is no longer effective.
  • The Discount End date field (F230) is a date field that contains the end date for the discount, i.e. the date at which the discount is no longer effective.
  • The Rebate End date field (F235) is a date field that contains the end date for the rebate, i.e. the date at which the rebate is no longer effective.

The Break/quantity End date field (F219) is a date field that contains the end date for the break/quantity, i.e. the date at which the break/quantity is no longer effective.
The Break/quantity is not exactly considered as a cost event since the required quantity may not be purchased to claim the reduction. The Break/quantity will give a reduction if you use the Inventory module to receive goods.

The Allow, Discount and Rebate cost events must contain valid Start and End dates for the eSuite software to recognize the cost reductions. If the end date for the reduction is not knon, enter a distant future date.
The Break/quantity does not require Start and End dates.

The Off Invoice (OI) field

The Off Invoice (OI) field contains the identification character of the off invoice status. This status specifies when the cost event value is given.

  • The Allow OI field contains the identification character of the off invoice status.
    This status specifies when the allowance value is given.
  • The Discount OI field contains the identification character of the off invoice status.
    This status specifies when the discount value is given.
  • The Rebate OI field contains the identification character of the off invoice status.
    This status specifies when the rebate value is given.
  • The Break/quantity OI field contains the identification character of the off invoice status.
    This status specifies when the break/quantity value is given.

The following status options are available:

OFF INVOICE STATUS VALUE DESCRIPTION
Off invoice 1 The cost event value is subtracted from the invoice total.
This affects the Next cost and Cost values in the Cost table, except for the Break/quantity cost event.
End of promotion E The cost event value is given at the end of the promotion (as a check) according to the number of items bought.
This does not affect the Next cost or Cost values in the Cost table.
On statement S The cost event value appears on the vendor's statement. It is given according to the number of items bought.
This does not affect the Next cost or Cost values in the Cost table.
Movement required M The cost event value is given (as a check) according to the number of items sold.
This does not affect the Next cost and Cost values in the Cost table.
